## 一、现存问题
32位地址空间, 4k页大小(512=2^9, 4096=2^12)，代表12位偏移量，20位页号。
一个页表项4字节(数组结构，每项包含页框号20位和一些控制位)
+---------------------------+--------+
|   物理页框号 (20 bit)      | 控制位  |
+---------------------------+--------+
此时因为每个进程都有独立页表，100个活动进程就有400MB

## 二、解决方案
### 1.简单的解决方案：更大的页
案例：
以32位地址空间为例，但这次假设用16KB的页。因此，会有18位的VPN加上14位的偏移量。假设每个页表项（4字节）的大小相同，现在线性页表中有218个项，因此每个页表的总大小为1MB，页表缩到四分之一。

优势：
1. 大页在数据库管理系统和其他高端商业应用常见。但主要目的是为了减少TLB的压力，防止未命中。然而采用多种页大小使得操作系统虚拟内存管理程序更复杂，暴露给程序新接口，让其请求大内存页更容易

劣势：
1. 造成内部碎片

### 2.混合方法：分页和分段
看着复杂又没有实际应用，先跳过。主流使用多级页表

### 3.多级页表
本质就是数组套数组

页目录（1024项，只占4KB）
┌─────────────────────────────┐
│ 项0  → 二级页表在物理页 50  │  ← 有内容
│ 项1  → 空                   │
│ 项2  → 空                   │
│ 项3  → 空                   │
│ ...  （全是空）              │
│ 项1023 → 空                 │
└─────────────────────────────┘

二级页表（只分配了1张，4KB，在物理页50处）
┌─────────────────────────────┐
│ 项0  → 物理页框号 100       │
│ 项1  → 物理页框号 203       │
│ 项2  → 物理页框号 57        │
│ 项3  → 空                   │
│ ...                         │
└─────────────────────────────┘

### 5.将页表交换到磁盘

最后，我们讨论放松最后一个假设。到目前为止，我们一直假设页表位于内核拥有的物理内存中。即使我们有很多技巧来减小页表的大小，但是它仍然有可能是太大而无法一次装入内存。因此，一些系统将这样的页表放入内核虚拟内存（kernel virtual memory）​，从而允许系统在内存压力较大时，将这些页表中的一部分交换（swap）到磁盘。
