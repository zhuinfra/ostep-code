## 一、分页的由来
操作系统解决空间管理问题的两种方法
1.分段：容易碎片化，随着时间推移分配内存变得困难
2.分页：将进程的虚拟地址空间分成固定大小的页，将物理内存也分成固定大小的页帧(物理内存)。每个页都有一个页表项，记录页帧的地址。

分页的优点
1. 灵活性，不假定堆和栈的增长方向以及如何使用。
2. 简单

## 二、虚拟页与页帧

虚拟页 = 程序看到的页（假的）
页帧   = 物理内存条上的页（真的）
页表   = 两者的映射关系（翻译表）

虚拟页与页帧：
虚拟页：4KB（固定，软件定义）
页帧 ：4KB（固定，硬件定义）
大小必须一致，否则没法一一对应
数量不同：
虚拟页数量：取决于虚拟地址空间大小
  32位系统：4GB ÷ 4KB = 约 100 万个虚拟页
物理页帧数量：取决于实际内存条
  16G 内存：16GB ÷ 4KB = 约 400 万个页帧
虚拟页可以比页帧多 → 多出来的部分映射到 swap


## 三、页表
为了记录地址空间的每个虚拟页对应的页帧地址，操作系统为每个进程维护一个页表。
页表项包括虚拟页号和页帧号。

虚拟地址 = 页号 * 页大小 + 偏移量
假设地址空间64字节，页大小16字节，也就是总共有4个虚拟页
高2位是页号可以表示4个虚拟页，低4位是偏移量可以囊括一个完整页。


## 四、页表存哪里
32位地址空间，4KB页大小：
虚拟地址 = 32位
  ├── 高20位 = VPN（虚拟页号）→ 查页表用
  └── 低12位 = 偏移量        → 页内位置
为什么是20位和12位？
  4KB = 2^12 字节 → 偏移量需要 12 位
  剩下的 32 - 12 = 20 位 → VPN
每个 PTE 占 4 字节
  （存物理页帧号 + 权限位 + 存在位 + 脏位等）
整个页表大小：
  2^20 × 4字节 = 4,194,304 字节 = 4MB
页表就像一个有100万行的表格：
虚拟页号      PTE内容（4字节）
─────────────────────────────
  0       →  物理页帧5234 + 权限标志
  1       →  物理页帧8921 + 权限标志
  2       →  在 swap 中
  3       →  未分配
  ...
  999999  →  物理页帧127 + 权限标志
100万行 × 每行4字节 = 400MB
由于页表如此之大，我们没有在MMU中利用任何特殊的片上硬件，来存储当前正在运行的进程的页表，而是将每个进程的页表存储在内存中。
```
# 查看页表占用的物理内存
cat /proc/meminfo | grep PageTables
```


## 五、页表中究竟有什么？
页表是一种数据结构，目的是将虚拟页号映射到物理页帧号。任何数据结构都可使用。此处使用数组，后续会有更高级的数据结构。
每个PTE有许多不同的位，例如：
有效位（Valid Bit）：
  表示该页表项是否有效，即该虚拟页是否已被分配物理页帧。
保护位（Protection Bits）：
  包括读取、写入和执行权限。
存在位（Present Bit）：
  表示该页是否当前在物理内存中。
脏位（Dirty Bit）：
  表示该页被带入内存后是否被修改过。
参考位（Reference Bit）：
  用于实现页面替换算法，例如最近最少使用（LRU）。
通过阅读英特尔架构手册，获取有关x86分页支持的更多详细信息。

## 六、分页：也很慢
```
movl 21, %eax
```
系统必须把虚拟地址21转为物理地址，操作系统必须先从进程的页表中提取页表项，执行转换，然后从物理内存中加载数据。
1. 假设页表基址寄存器包含页表的起始位置的物理地址。算找到页表
2. 硬件通过&运算得出虚拟页, 该值作为页表基址寄存器指向PTE数组的索引。
3. 校验有效位和保护位
4. 通过偏移量计算物理地址
具体看书籍代码更清晰。
两个问题
1. 慢
2. 占用太多内存
